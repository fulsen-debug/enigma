1) Product + Integration Scope

KOBECOIN AI Guardian is a Solana risk-intelligence layer designed for traders and trading platforms.

  • Pre-entry token behavior analysis (kill-switch, concentration, authorities, connected holders)
  • One-token scanner workflow for fast discretionary review
  • Holder behavior table for deeper due diligence
  • Paper-agent execution architecture for one selected token

Boundary: This first release is paper-first. Public live execution is not part of the shipped product surface.

3) Authentication Flow (Wallet + JWT)

  1. Create nonce: POST /api/auth/nonce
  2. Wallet signs nonce message
  3. Verify signature: POST /api/auth/verify
  4. Server sets an HttpOnly session cookie for the browser app
  5. Bearer tokens remain supported for scripts and QA tooling
curl -X POST https://your-domain.com/api/auth/nonce \
  -H "content-type: application/json" \
  -d '{"wallet":"<WALLET_ADDRESS>"}'
curl -X POST https://your-domain.com/api/auth/verify \
  -H "content-type: application/json" \
  -d '{"wallet":"<WALLET_ADDRESS>","nonce":"<NONCE>","signature":"<BASE58_SIGNATURE>"}'

5) Trader App Integration Pattern

  1. User opens token page (Pump.fun or your own trading UI).
  2. Client calls /api/signal before entry.
  3. Policy engine decides: allow / caution / block.
  4. If allowed, trader executes via exchange/router (Jupiter Ultra or other).
  5. Post-entry, client re-checks the token with /api/signal and /api/token/holders as needed for risk drift.

This preserves control for the trader while enabling semi-automated risk handling.

7) Report Generation Integration

If another product needs AIG-style reports, use the scanner JSON as the source document and map these blocks directly:

  • signal.token: identity, symbol, image
  • signal.market: price, liquidity, volume, flow
  • signal.killSwitch: status, score, risk narrative
  • signal.marketRegime: volatility, ADX, regime label
  • signal.tradePlan: entry zone, support, stop, targets
  • signal.rugPullRisk: bundle risk and forensic summary
const report = {
  mint,
  symbol: scan.signal.token.symbol,
  status: scan.signal.status,
  patternScore: scan.signal.patternScore,
  confidence: scan.signal.confidence,
  market: scan.signal.market,
  tradePlan: scan.signal.tradePlan,
  forensics: scan.signal.rugPullRisk
};

The web app uses this same payload to render the dashboard card, PNG export, and X share summary.

9) Jupiter Ultra Execution Bridge

Recommended execution chain:

  1. Risk Gate: Kobecoin AI Guardian scan + policy decision
  2. Quote/Order: Jupiter /ultra/v1/order
  3. Signing: user wallet signs base64 tx
  4. Execute: Jupiter /ultra/v1/execute
  5. Monitor: Kobecoin AI Guardian scanner loop + alerts

Best practice: never execute when Kobecoin AI Guardian verdict is HIGH_RISK unless trader explicitly overrides.

15) Auto-Trade Mode (Policy Controlled)

Auto-trade is implemented as a policy engine with safe defaults.

  • Agent loop is single-token mode: pass one mint per run/tick request.
  • Watchlist endpoints remain scanner-only.
  • Default mode is paper and disabled
  • Engine tick supports layered entries on the same agent mint, bounded by maxOpenPositions and cooldownSec.
  • Per-user execution locks prevent overlapping /run, /engine/tick, and /monitor calls from opening duplicate concurrent cycles.
  • /api/autotrade/* has dedicated rate limiting for loop safety under high-frequency polling.
  • allowCautionEntries=true enables faster entries by accepting CAUTION status (HIGH_RISK remains blocked).
  • allowHighRiskEntries=true can force entries even in HIGH_RISK state (extreme risk).
  • minPatternScore can be set to 0 to disable pattern gate in policy checks.
  • Only FAVORABLE signals that pass thresholds become buy candidates
  • Live mode can execute Jupiter swaps when execution env + signer are configured
  • Dashboard trading UI maps directly to these endpoints: Test Mode (Paper) -> /api/autotrade/config and /api/autotrade/run, Live Mode (Agent) -> /api/autotrade/execution-config, /api/autotrade/engine/tick, and /api/autotrade/monitor
  • UI guardrails normalize invalid exits automatically (SL and trailingStopPct remain below tpPct) before saving execution config
  • Run history retention is capped by ENIGMA_AUTOTRADE_RUN_RETENTION (default 1200) to avoid unbounded log growth.
curl -X PUT https://your-domain.com/api/autotrade/config \
  -H "authorization: Bearer <JWT>" \
  -H "content-type: application/json" \
  -d '{
    "enabled": true,
    "mode": "paper",
    "allowCautionEntries": false,
    "minPatternScore": 74,
    "minConfidence": 0.78,
    "maxConnectedHolderPct": 20,

14) Market Regime Tuning Notes

  • ADX period: default 14. Increase to smooth noise, decrease for faster response.
  • Volatility window N: ATR percentile uses last 200 ATR samples by default.
  • Regime thresholds: defaults are ADX 25 for trending and VolatilityIndex 60 for expanding volatility.
  • Cache TTL: candle cache defaults to ENIGMA_CANDLE_CACHE_TTL_SEC=60 to avoid recalculating every render.
  • Regime cache TTL: market regime summary cache defaults to ENIGMA_MARKET_REGIME_CACHE_TTL_SEC=45.
  • Timeframes supported: 5m, 15m, 1h, 4h, 1d; scanner prefers 1h and falls back automatically when data is insufficient.

15) Agent Execution (Deterministic Path)

  • Decision path: execution uses deterministic numeric rules only (no chat/LLM dependency in trade open/close decisions).
  • Regime policy matrix: combines ADX, VolatilityIndex, liquidity, participation, estimated spread, and holder risk to switch style: BREAKOUT, TREND, MEAN_REVERT, NO_TRADE.
  • Adaptive entry triggers: pullback in buy zone, momentum continuation above zone, or timeout fallback with reduced size.
  • Dynamic size formula: size = min(maxRiskPerTrade / stopDistance, maxPositionCap), then scaled by volatility/spread/regime multipliers.
  • Risk hard limits: trade/day/hour/token loss caps, loss-streak cooldown pause, max concurrent positions, and hostile-regime kill-switch.
  • Event-driven exit watcher: monitor endpoint supports fast multi-sweep checks to evaluate TP/SL/trailing/timeout exits between entry scans.
  • Protective orders: use exchange/router-native protective order flow when venue supports it; current default path is deterministic fast-poll fallback.
  • PnL attribution: open/close actions include expected vs realized and execution drag (fees/slippage/latency/partial-fill effects).
  • Tuning env vars: ENIGMA_SAFETY_MAX_HOURLY_LOSS_USD, ENIGMA_SAFETY_MAX_TOKEN_DAILY_LOSS_USD, ENIGMA_SAFETY_MAX_TOKEN_HOURLY_LOSS_USD, ENIGMA_SAFETY_MAX_LOSS_PER_TRADE_USD, ENIGMA_SAFETY_LOSS_STREAK_THRESHOLD, ENIGMA_SAFETY_LOSS_STREAK_PAUSE_SEC, ENIGMA_SAFETY_EXIT_ON_HOSTILE_REGIME.
